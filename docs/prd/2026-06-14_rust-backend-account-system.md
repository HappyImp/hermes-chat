# PRD: Rust 后端 + 账户系统

## 1. 背景

当前 hermes-chat 前端直连 Hermes Gateway，无账户系统，所有用户共享会话。存在以下问题：

- 会话数据存 localStorage，换设备丢失
- 无法区分不同用户的权限
- 无法限制员工访问权限
- 无安全防护，聊天内容可直接攻击服务器

## 2. 目标

### 2.1 核心目标

| 目标 | 描述 |
|------|------|
| 会话持久化 | 会话数据存后端数据库，跨设备同步 |
| 账户隔离 | 各账户会话独立，互不可见 |
| 员工权限 | 各账户可使用的员工独立配置 |
| 安全防护 | 输入验证、权限检查、防注入 |

### 2.2 技术目标

| 指标 | 目标值 |
|------|--------|
| 并发用户 | ≥ 50 |
| 聊天延迟 | ≤ 100ms（不含 LLM 响应） |
| 数据库 | SQLite（单机部署） |
| 部署方式 | Docker / systemd |

## 3. 功能需求

### 3.1 账户系统

#### 3.1.1 用户注册
- 用户名：3-32 字符，字母数字下划线
- 密码：8-128 字符，至少包含大小写字母和数字
- 默认角色：user

#### 3.1.2 用户登录
- 用户名 + 密码登录
- 返回 JWT token（有效期 24 小时）
- 支持 token 刷新

#### 3.1.3 用户角色
| 角色 | 权限 |
|------|------|
| admin | 全部权限，可管理用户和员工分配 |
| user | 使用分配的员工，管理自己的会话 |

### 3.2 会话管理

#### 3.2.1 会话 CRUD
- 创建会话（关联账户）
- 查询会话列表（按账户过滤）
- 删除会话（软删除）
- 会话标题自动/手动更新

#### 3.2.2 消息管理
- 消息持久化（存数据库）
- 消息历史查询
- 消息分页加载

### 3.3 聊天代理

#### 3.3.1 SSE 流式代理
- 前端 → Rust 后端 → Hermes Gateway
- 透传 SSE 流式响应
- 支持中断/取消

#### 3.3.2 权限检查
- 验证 JWT token
- 检查用户是否有权使用指定员工
- 记录聊天日志

### 3.4 员工权限

#### 3.4.1 员工分配
- 管理员可为用户分配可用员工
- 用户只能与分配的员工对话
- 默认员工列表可配置

#### 3.4.2 员工状态
- 代理 Hermes API 获取员工状态
- 按用户权限过滤显示

### 3.5 安全防护

#### 3.5.1 输入验证
- 消息长度限制（≤ 10000 字符）
- 消息频率限制（每用户 10 条/分钟）
- 特殊字符过滤

#### 3.5.2 权限检查
- JWT 验证
- RBAC 权限控制
- 会话所有权验证

#### 3.5.3 防攻击
- SQL 注入防护（参数化查询）
- XSS 防护（输出编码）
- CSRF 防护（SameSite cookie）
- 速率限制（IP + 用户双重限制）

## 4. 非功能需求

### 4.1 性能
- API 响应时间 ≤ 100ms（不含 LLM）
- 支持 50 并发用户
- SQLite 读写分离（WAL 模式）

### 4.2 可靠性
- 数据库自动备份
- 优雅降级（Hermes 不可用时返回错误）
- 日志记录（访问日志、错误日志）

### 4.3 可维护性
- 代码结构清晰
- 配置外部化
- Docker 部署支持

## 5. 技术约束

- Rust 后端必须与现有 Hermes Gateway 兼容
- 前端改动最小化（主要改 API 调用路径）
- 数据库使用 SQLite（单机部署简单）

## 6. 验收标准

- [ ] 用户可注册、登录、获取 token
- [ ] 用户会话数据持久化，跨设备同步
- [ ] 用户只能看到自己的会话
- [ ] 管理员可分配员工权限
- [ ] 用户只能与分配的员工对话
- [ ] 聊天 SSE 流式响应正常
- [ ] 输入验证和安全防护生效
- [ ] 性能指标达标
