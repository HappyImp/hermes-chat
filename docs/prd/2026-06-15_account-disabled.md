# PRD: 账户禁用功能

## 1. 背景

当前 hermes-chat 账户系统缺少账户状态管理功能。管理员无法临时禁用用户账户，只能删除用户。这在实际运营中存在以下问题：

- 用户违规时无法临时封禁，只能删除账户（数据丢失）
- 无法实现账户安全保护（如检测到异常登录时自动禁用）
- 缺少账户生命周期管理能力

## 2. 目标

### 2.1 核心目标

| 目标 | 描述 |
|------|------|
| 账户状态控制 | 管理员可启用/禁用用户账户 |
| 访问拦截 | 被禁用账户无法访问系统（JWT 验证后拦截） |
| 状态持久化 | 账户状态存储在数据库，重启后保持 |

### 2.2 技术目标

| 指标 | 目标值 |
|------|--------|
| 拦截位置 | 认证中间件（JWT 验证后、业务逻辑前） |
| 错误码 | HTTP 403 Forbidden |
| 响应格式 | `{"error": "账户已被禁用，请联系管理员"}` |

## 3. 功能需求

### 3.1 数据模型

#### 3.1.1 users 表扩展
- 新增 `enabled` 字段（INTEGER，1=启用，0=禁用）
- 默认值：1（启用）

### 3.2 管理员 API

#### 3.2.1 禁用账户
- 管理员可禁用指定用户
- 禁用后该用户无法登录和访问 API
- 已登录用户的 token 在下次请求时被拦截

#### 3.2.2 启用账户
- 管理员可启用被禁用的用户
- 启用后用户可正常登录和访问

### 3.3 认证中间件

#### 3.3.1 账户状态检查
- JWT 验证通过后，查询用户 `enabled` 字段
- 如果 `enabled = 0`，返回 HTTP 403 错误
- 错误信息：`账户已被禁用，请联系管理员`

### 3.4 前端处理

#### 3.4.1 错误展示
- 前端统一读取 `data.error` 字段
- 展示后端返回的错误信息
- 不再 fallback 到 `data.message`

## 4. 非功能需求

### 4.1 性能
- 账户状态查询应使用索引
- 不应显著增加认证延迟

### 4.2 安全性
- 只有管理员可修改账户状态
- 禁用操作记录审计日志
- 前端隐藏被禁用用户的操作入口

### 4.3 可维护性
- 错误处理统一使用 `AppError::Auth(AuthError::AccountDisabled)` 枚举
- 避免重复的错误枚举定义

## 5. 技术约束

- 数据库使用 SQLite，`enabled` 字段为 INTEGER 类型
- 后端使用 Rust axum 框架
- 前端使用 Vue + TypeScript
- 错误响应格式统一为 `{"error": "..."}`

## 6. 验收标准

- [ ] 管理员可禁用用户账户
- [ ] 管理员可启用被禁用用户账户
- [ ] 被禁用用户无法登录（返回 403）
- [ ] 被禁用用户已登录 token 被拦截（返回 403）
- [ ] 启用后用户可正常登录和访问
- [ ] 错误响应格式统一为 `{"error": "..."}`
- [ ] 前端正确展示禁用错误信息
- [ ] 账户状态持久化（重启后保持）
